## 1. HOSTILE-JUDGE ATTACK SCENARIOS

1. **[BLOCKER] A judge can break the "Docker Compose hash + signer address" story by asking for the missing binding, not by debating hashes.** The three 60-second questions are: (1) "Show me the quote field that contains the receipt signing public key or challenge"; (2) "Show me the code path proving the key that signed this receipt was generated only after that attested container booted"; (3) "Show me why a copied signer key in a same-hash or stale-hash deployment cannot mint receipts." The original 5b language named the missing REPORTDATA binding but did not fully explain that this proves a known 0G-attested service identity, not an enclave-born Compass receipt key. The strengthened 5b now says that explicitly, but the underlying gap remains. That distinction matters because `AgentRegistry.verifyAttestation` is only a non-empty-byte stub, and the receipt schema still says the verifier public key "MUST be bound into the TEE attestation REPORTDATA." Evidence: `docs/honest-limits.md:40`, `docs/honest-limits.md:49`, `docs/honest-limits.md:54`, `contracts/contracts/AgentRegistry.sol:110`, `contracts/contracts/AgentRegistry.sol:126`, `docs/schemas/receipt-v1.json:55`.

2. **[HIGH] The fastest hostile attack is to request a fresh verification transcript, then point out that none of the checks prove policy-evaluator execution.** Live 0G docs expose `verifyService` checks for TEE signer address and Docker Compose hash, and say full verification still needs manual dstack/image-integrity steps; `processResponse(providerAddress, chatID)` verifies response integrity for a provider chat ID, not arbitrary Compass JSON. In repo terms, the old architecture still claims receipt hash and enclave pubkey are embedded into `processResponse` content and then bound into REPORTDATA; that is now known false for the committed verdict. Evidence: `docs/architecture.md:88`, `docs/architecture.md:89`, `docs/architecture.md:90`, `docs/notes/0g-ecosystem-status.md:57`; live 0G docs: `https://docs.0g.ai/developer-hub/building-on-0g/compute-network/inference` (`verifyService`, `processResponse` sections).

3. **[MEDIUM] The privacy claim survives only if it is narrowed to bounded disclosure, not enclave-origin proof.** The claim "Compass never puts raw credentials on-chain and emits only a non-identifying receipt" is still coherent because `ReceiptIssued` contains hashes, nullifier, expiry, attestation digest, and timestamp bucket rather than identity fields. The claim "the receipt was necessarily produced by Compass policy code inside a TEE" is not coherent under implicit key trust. The honest-limits language must say the gap does not tank the privacy claim because the disclosed output remains non-identifying, but it does tank formal proof that the signer key and policy evaluator were enclave-born. ## 6. OPTION (iv) DISMISSAL — WAS IT TOO QUICK?

1. **[BLOCKER] If TeeML cannot run Compass's deterministic evaluator inside the measured runtime, the marginal security gain over hardcoded ECDSA is near zero.** With no REPORTDATA binding, the key security claim becomes "we trust a signer address associated with a 0G-verified compose hash." A plain Node.js signer can also publish a Docker Compose hash and expected signer address; the only meaningful extra from TeeML is independent 0G attestation of a provider environment. That extra matters only if the environment actually contains the Compass evaluator and receipt signer. If it only signs model completions or routes inference, option ii has the schedule risk of TEE integration without the core receipt-origin proof. Evidence: `docs/architecture.md:88`, `docs/architecture.md:90`, `docs/honest-limits.md:46`, `docs/honest-limits.md:54`, live 0G docs `https://docs.0g.ai/developer-hub/building-on-0g/compute-network/inference`.

2. **[HIGH] If custom evaluator execution is confirmed, the marginal gain is real but narrow.** The gain would be: a judge can verify a 0G-attested provider identity and compose hash, plus a receipt signature from the expected key, while Compass honestly admits the key is not hardware-quote-bound. That is stronger than plain Node.js because the runtime identity is externally attested by 0G, but weaker than REPORTDATA platforms because key substitution cannot be ruled out cryptographically. Evidence: `docs/honest-limits.md:49`, `docs/honest-limits.md:50`, `docs/honest-limits.md:54`, `docs/notes/0g-ecosystem-status.md:61`.

3. **[MEDIUM] The option iv dismissal should be reframed as a business/judging decision, not a security decision.** Pure-0G Plan B is justified if Track 5 rewards visible 0G-native integration and the demo can be made honest. It is not justified by a large cryptographic advantage unless custom measured Compass code is proven. The verdict should explicitly say "we accept weaker formal proof to maximize 0G-native integration under the hackathon clock." Evidence: `docs/notes/0g-ecosystem-status.md:47`, `docs/notes/0g-ecosystem-status.md:73`, `docs/notes/0g-ecosystem-status.md:75`.

## Net verdict on the verdict

The committed verdict still holds as a pragmatic Track 5 bet if, and only if, 0G TeeML can run Compass's deterministic policy evaluator and receipt signer inside the measured runtime. The schedule estimate is optimistic by one failure mode: a broken compute SDK is not a small contingency if it forces direct-RPC integration. The honest-limits language must be strengthened so it preserves the bounded-disclosure privacy claim while admitting that formal enclave-born-key proof is absent. Immediate remediation is to get a reproducible `verifyService`/manual quote transcript, confirm whether custom evaluator code is actually measured by TeeML, and set a Day-10 rather than Day-15 trigger for any non-0G reportData escape hatch.
