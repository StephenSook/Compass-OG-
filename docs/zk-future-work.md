# ZK future-work proposal

Compass v1 uses **SD-JWT VC selective disclosure + sealed TEE
inference** instead of a zero-knowledge proof of eligibility. This
note explains the trade-off, why v1 chose the path it did, and what a
ZK variant would look like if the project takes that direction in v2+.

## What v1 does today

1. The user holds an SD-JWT VC issued by an NGO. The VC commits to
   claims `{name, hkid, visa_status, employer, …}` via a per-claim
   salted hash table.
2. The user discloses only the claims relevant to the policy
   (`{visa_status, age >= 18, hk_resident}`), withholding the rest.
3. A sealed receipt-signer inside Phala dstack TDX evaluates the
   policy predicate against the disclosed claims and signs the
   eligibility receipt.
4. The on-chain `ReceiptIssued` event commits to the policy hash, a
   nullifier (preventing replay), an agent identity commitment, and
   a 15-minute timestamp bucket. No name, no HKID, no employer.

The trust model is: "we trust the TEE attestation chain — the per-receipt
RA quote binds `(signer, image, receiptId)`, the verifier checks the
quote against the public report-data, and the
composeHash gives a publicly-verifiable handle to the *exact image*
that signed the receipt." A subpoena reaches the bucketed timestamp
and the commitment, never the disclosed claims.

## Where v1 falls short

- **TEE trust assumption.** TDX is a strong assumption but not
  unbreakable; Intel SGX has had publicly-disclosed side-channel
  attacks before, and TDX research is still maturing. Compass leans
  on the per-receipt quote binding to defeat the most pragmatic
  attack vectors, but a sufficiently-resourced adversary with TEE
  zero-day capability would break the v1 guarantee.
- **Disclosed claims still cross the TDX boundary.** A worker
  technically discloses her visa status and DOB to the sealed
  receipt-signer — those bytes leave the worker's device. We trust
  them to stay inside the TEE; we cannot prove they do.

## What a ZK variant would look like

A v2 with a Groth16- or Plonky2-style proof of eligibility would shift
the trust model from "trust the TEE attestation" to "trust the
zero-knowledge soundness of the SNARK." Sketch:

```
predicate ZKEligibilityPredicate(
  attrs:          Claims,         // private witness
  policyDigest:   bytes32,        // public input
  policyParams:   PolicyParams,   // public input
  receiptId:      bytes32,        // public input
  nullifier:      bytes32         // public input (derived from claims)
)
  // Decomposed proof obligations:
  // 1. Claims signed by a trusted issuer (verify Ed25519 inside the SNARK).
  // 2. policyDigest == keccak256(policyParams).
  // 3. Predicate(attrs, policyParams) == true (the eligibility check).
  // 4. nullifier == HashWithSalt(attrs.identifier, policyDigest).
  // 5. receiptId binds to (nullifier, blockTimestamp, policyDigest).
```

The on-chain verifier would be a Solidity `Verifier.sol` autogenerated
from the SNARK toolchain (snarkjs / arkworks / halo2). The
`CompassHub.consumeGrantAndIssueReceipt` flow would replace the EIP-712
grant + TEE attestation digest with a SNARK proof.

### What's hard about a ZK variant

1. **Verifying Ed25519 inside a SNARK is expensive.** A typical
   constraint count for Ed25519 verification is ~100k-500k R1CS
   constraints. With Groth16 over BN254 that means a proving time of
   ~5-30 seconds in WASM, ~1-3 seconds native. Acceptable for
   batch use cases; brutal for an interactive intake flow where the
   worker is at the kiosk with a clinician.
2. **Policy circuits need a trusted-setup ceremony per policy.** If
   each NGO has its own policy circuit, that's a multi-policy
   ceremony per registered policy. Universal-setup proof systems
   (Plonky2, Halo2) avoid this but trade off proof size and
   verification cost.
3. **Browser-side proving requires WASM tooling that is still
   maturing.** snarkjs works; arkworks WASM is improving;
   halo2-wasm is preliminary.
4. **The SD-JWT VC format is selective-disclosure-friendly but not
   SNARK-friendly.** A SNARK-friendly issuance scheme (BBS+
   signatures, or Anonymous Credentials Light) shifts the
   issuer-side complexity. NGOs would need an issuer key in the
   BBS+ scheme — which is fine for v2 but adds an integration step.

### What the trade-off would buy

The TEE assumption is *removed*. The claims literally never leave the
user's device — only the proof does. A government-level adversary
that compromises Phala dstack TDX still cannot decrypt the
worker's claims.

The trade is: longer proving time at the kiosk, a new issuer-side
crypto path, and a per-policy circuit + setup ceremony.

## Recommendation

**Compass v2 should add a ZK *option*, not a ZK *replacement***. The
SD-JWT + TEE path is the right default for the migrant-worker context
because it's interactive-grade fast (sub-second receipt mint) and
because the threat model — abusive employer with subpoena power, not
a nation-state — matches the TEE guarantee.

For the *small subset* of cases where the TEE assumption is
insufficient (e.g., a worker fleeing a government as well as an
employer), an optional ZK path lets the system support both. The user
chooses at intake which path applies.

## When to revisit

- After the SD-JWT VC standard finalises (`draft-ietf-oauth-sd-jwt-vc`
  → RFC). Probably 2026 Q3-Q4.
- After Plonky3 or Halo2 stabilise their WASM proving experience.
- If TDX has a publicly-disclosed exploit that materially weakens
  the per-receipt quote binding. Monitor the Intel TDX security
  advisories list.

## Open questions

- Could the per-policy circuit be *parameterized* (single circuit, take
  policy params as public input) to avoid per-policy ceremonies?
  Possibly with Plonky2 + parameterized lookup tables.
- Could the nullifier be computed *outside* the SNARK using a
  homomorphic commitment, reducing the in-circuit work? Maybe via
  Pedersen commitments + a Schnorr-style adapter.
- What's the right reference application to benchmark against? The
  Privacy Pools / Tornado Cash v2 work has the closest performance
  envelope.
